## 3. 背景、问题定义和产品目标

### 3.1 背景

中国历史人物、事件、朝代、关系链天然适合可视化探索，但常见历史内容产品多以文章列表、词条页或线性时间轴呈现，探索成本高，横向比较弱，用户难以快速建立“人物在什么朝代、与谁相关、卷入了哪些事件、处在怎样的权力网络里”的认知结构。

### 3.2 问题定义

目标不是再做一个历史百科站，而是做一个“能浏览、能搜索、能钻取、能建立关联认知”的历史关系图谱产品。

需要解决三个核心问题：

1. 用户如何快速进入一个朝代，并理解该朝代的核心人物结构。
2. 用户如何从任意人物或事件出发，沿着关系不断探索上下文。
3. 用户如何在“大量历史信息”与“清晰页面体验”之间找到平衡，而不被纯文本淹没。

### 3.3 产品目标

一期目标：

1. 建立一个以朝代为组织骨架、以帝王为关系中心的图谱浏览产品。
2. 提供以人物、事件、朝代为入口的搜索与跳转能力。
3. 通过图谱、时间轴、详情信息和插图组合，形成沉浸式浏览体验。
4. 为后续持续补充数据提供稳定的页面壳和交互结构。

## 4. 成功指标和结果衡量方式

### 4.1 产品结果指标

1. 用户进入任一朝代后，可在 30 秒内理解核心帝王、关键人物和主要事件的基本关系。
2. 用户从一次搜索进入后，可在 3 步内到达目标人物或事件的关系上下文。
3. 任一人物详情页都可继续延展到相关人物和相关事件，不形成死页面。
4. 产品在历史数据尚不完整时，页面仍能正常工作，不因缺少资料而失去主路径。

### 4.2 一期验收指标

1. 支持人物、事件、朝代三类搜索。
2. 支持按朝代切换图谱，并从帝王节点逐层展开。
3. 支持图谱平移、缩放、节点点击、层级展开和上下文高亮。
4. 支持人物详情、事件详情、朝代概览三类信息面板。
5. 支持插图位和关系卡片位，即使内容未填满也能按统一结构展示。

## 5. 关键产品决策与规划前提

### 5.1 已采纳决策

1. 产品形态：一期定义为 Web 端探索型产品，不做纯文章站。
2. 主浏览模式：以“朝代视角进入 + 图谱视角探索 + 搜索直达定位”为主。
3. 图谱结构：每个朝代以帝王为核心节点，向宗室、重臣、名将、谋士、后妃、敌对势力、关键事件逐层延展。
4. 搜索策略：搜索命中后优先进入其所处历史上下文，而不是直接落到孤立详情页。
5. 内容策略：一期优先建立结构和交互壳，允许节点信息不完整，但不允许交互路径断裂。
6. 表达策略：图谱不是唯一视图，必须与时间轴、详情信息、插图共同组成复合页面。

### 5.2 规划前提

1. 一期不要求完整覆盖所有历史人物数据，但产品结构必须支持从最早可追溯朝代开始逐步补齐。
2. 一期以公开浏览为主，不考虑登录、收藏、UGC、评论、多人协作编辑。
3. 一期仅定义浏览器可访问的 Web 端体验，不考虑移动端和客户端形态。

## 6. 范围、非目标、延后项和优先级分层

### 6.1 P0 范围

1. 首页 / 进入页
2. 朝代选择与朝代概览
3. 图谱主画布
4. 搜索入口与搜索结果跳转
5. 人物详情面板
6. 事件详情面板
7. 时间轴辅助区
8. 插图展示位与信息卡片位
9. 空态、缺失态、加载态、无结果态

### 6.2 P1 延后项

1. 地图维度联动
2. 多人物对比模式
3. 自定义关系筛选器
4. 收藏、订阅、学习路径
5. 音频讲解、旁白、导览模式
6. 用户生成内容或纠错反馈机制

### 6.3 非目标

1. 不做史料考据工具。
2. 不做历史论文式长文阅读站。
3. 不做后台录入系统。
4. 不在一期解决全量历史数据标准化问题。
5. 不做复杂社交功能。

## 7. 目标用户、角色和使用前提

### 7.1 目标用户

1. 对中国历史有兴趣，但不想通过大段文本学习的泛历史用户。
2. 需要快速理清人物关系和事件脉络的学生、内容创作者、历史爱好者。
3. 对图谱探索、沉浸式知识体验感兴趣的可视化产品用户。

### 7.2 用户目标

1. 快速知道某个人物属于哪个朝代、与谁关系最强、参与了什么事件。
2. 从一个熟悉人物出发，探索其所在时代网络。
3. 用较低认知负担理解复杂历史关系。

### 7.3 使用前提

1. 用户无需登录即可浏览。
2. 用户默认具备基本搜索和图谱拖拽操作能力。
3. 当数据不完整时，系统仍需给出可理解的结构说明，而不是显示为异常。

## 8. 业务对象、核心状态和长期业务规则

### 8.1 业务对象

1. 朝代：产品的一级组织单元。
2. 帝王：朝代图谱的核心锚点。
3. 人物：除帝王外的重要角色，如宗室、臣子、将领、谋士、后妃、学者、敌对人物。
4. 事件：战争、政变、改革、继位、起义、外交、制度变迁等关键历史事件。
5. 关系：人物与人物、人物与事件、人物与朝代、事件与朝代之间的关联。
6. 插图：用于增强场景感与识别度的视觉内容单元。

### 8.2 对象状态

1. 已收录：对象具备基础资料，可展示在图谱和详情中。
2. 信息待补全：对象已存在，但部分字段缺失，需展示占位说明。
3. 关系待补全：对象可展示，但关系链尚不完整，需提示“关联持续补充中”。

### 8.3 长期业务规则

1. 任一人物必须至少归属一个朝代。
2. 任一事件必须至少关联一个朝代，且应尽量关联核心人物。
3. 任一帝王必须可作为进入其朝代图谱的核心入口。
4. 搜索命中的对象必须能落在一个明确上下文中，而不是只展示孤立卡片。
5. 数据不完整时允许节点存在，但必须明确提示缺失，而不能让用户误判为空白或故障。

## 9. 核心场景和关键任务路径

### 9.1 场景一：从朝代进入，理解时代结构

1. 用户进入首页。
2. 用户选择某一朝代。
3. 系统展示该朝代概览、核心帝王和主要关系簇。
4. 用户在图谱中点击帝王或关键人物。
5. 详情面板展示人物简介、角色身份、相关事件和重要关系。
6. 用户继续沿关系跳转，形成连续探索。

### 9.2 场景二：从搜索人物进入，快速定位上下文

1. 用户输入人物名称。
2. 系统返回人物结果，并标明所属朝代和身份标签。
3. 用户点击结果。
4. 系统进入对应朝代图谱，并自动定位到该人物节点。
5. 图谱高亮人物与一层核心关系，详情面板同时展开。

### 9.3 场景三：从事件进入，理解事件影响网络

1. 用户输入事件名称。
2. 系统返回事件结果。
3. 用户进入事件上下文。
4. 事件详情展示事件简介、发生时期、核心参与者和影响对象。
5. 图谱高亮相关人物与前后关键事件，帮助用户建立脉络。

## 10. 页面 / 模块范围和页面地图

### 10.1 页面地图

1. 首页 / 进入页
2. 朝代图谱页
3. 搜索结果层
4. 人物详情侧边面板
5. 事件详情侧边面板
6. 朝代概览面板

### 10.2 首页 / 进入页职责

1. 建立产品气质与主题认知。
2. 提供朝代入口、搜索入口和推荐探索入口。
3. 用主视觉插图或场景插图强化“历史关系图谱”而非普通知识站的第一印象。

### 10.3 朝代图谱页职责

1. 承载主图谱探索。
2. 展示当前朝代的核心人物网络与关键事件。
3. 提供时间轴辅助和上下文切换。

## 11. 功能需求

### 11.1 首页 / 进入页

#### 11.1.1 入口与信息组成

1. 顶部主搜索框。
2. 朝代时间长廊或朝代选择器。
3. 推荐入口区，如“从秦始皇开始”“从三国人物网络进入”“从玄武门之变进入”等。
4. 主视觉插图区。
5. 产品说明区，用简短语言告诉用户“可搜索人物、事件，并在图谱中持续探索关联”。

#### 11.1.2 交互要求

1. 点击朝代后进入该朝代图谱页。
2. 搜索命中后直接进入对应上下文。
3. 推荐入口应直接落到具体人物或事件上下文，不应跳到空白页。

### 11.2 朝代选择与切换

#### 11.2.1 功能要求

1. 支持按历史顺序浏览朝代。
2. 当前朝代需有明确高亮。
3. 切换朝代后，图谱和时间轴同步切换。
4. 若某朝代数据较少，也要展示基础骨架和“持续补充中”说明。

### 11.3 图谱主画布

#### 11.3.1 目标

让用户在单一主视图中理解“谁与谁相关、通过什么事件关联、当前朝代的权力和历史结构如何分布”。

#### 11.3.2 核心能力

1. 支持拖拽平移。
2. 支持缩放查看。
3. 支持点击节点查看详情。
4. 支持从帝王节点向外分层展开。
5. 支持高亮当前节点的一度关系。
6. 支持在用户点击事件节点后切换为事件关系高亮。
7. 支持关系密集时自动弱化非当前焦点信息，避免页面失控。

#### 11.3.3 信息分层原则

1. 帝王节点为一级视觉锚点。
2. 核心人物为二级重点。
3. 一般关联人物为三级信息。
4. 事件节点与人物节点在视觉语义上必须可区分。
5. 关系线需至少区分“血缘 / 政治 / 冲突 / 事件参与”四类语义层级。

#### 11.3.4 页面规则

1. 初次进入朝代图谱时，不一次性铺满所有节点，优先展示帝王、核心人物和核心事件。
2. 只有在用户主动展开或搜索定位后，才展示更深层关系。
3. 当画布信息过多时，优先保留当前焦点上下文，不要求保留所有细枝末节同屏展示。

### 11.4 搜索

#### 11.4.1 搜索对象

1. 人物
2. 事件
3. 朝代

#### 11.4.2 搜索体验

1. 支持联想输入。
2. 搜索结果中应显示对象类型、所属朝代和简短识别信息。
3. 人物重名时，必须显示朝代和身份标签辅助区分。
4. 搜索无结果时，提供“换一个人物 / 事件 / 朝代试试”的引导，不展示系统错误感。

#### 11.4.3 搜索跳转规则

1. 搜人物：进入所在朝代图谱并定位人物节点。
2. 搜事件：进入对应朝代图谱并高亮事件及关键参与者。
3. 搜朝代：进入该朝代概览与图谱首页态。

### 11.5 人物详情面板

#### 11.5.1 信息结构

1. 人物名称
2. 朝代与身份标签
3. 简介
4. 与当前图谱核心的关系说明
5. 相关事件列表
6. 关键关联人物列表
7. 插图位

#### 11.5.2 交互要求

1. 点击相关人物可重定位图谱焦点。
2. 点击相关事件可切换到事件关系视图。
3. 当人物信息不完整时，仍展示现有结构与占位说明。

### 11.6 事件详情面板

#### 11.6.1 信息结构

1. 事件名称
2. 发生时期
3. 所属朝代
4. 事件简介
5. 核心参与人物
6. 前后关联事件
7. 插图位或事件场景图位

#### 11.6.2 交互要求

1. 用户可从事件回到人物视角。
2. 用户可沿着前后关联事件继续探索。

### 11.7 朝代概览面板

#### 11.7.1 信息结构

1. 朝代名称
2. 时间区间
3. 朝代一句话概述
4. 核心帝王列表
5. 代表事件列表
6. 推荐探索入口

### 11.8 时间轴辅助区

#### 11.8.1 目标

避免用户只看到空间关系而看不到时间顺序。

#### 11.8.2 功能要求

1. 展示当前朝代的关键事件时间序列。
2. 点击时间轴事件，可联动高亮图谱。
3. 当前焦点人物相关的关键事件应在时间轴中有明显标记。

### 11.9 插图策略

#### 11.9.1 插图职责

1. 增强历史氛围与记忆点。
2. 帮助用户快速识别人物和事件类型。
3. 缓和纯图谱与纯文字的认知疲劳。

#### 11.9.2 插图使用位置

1. 首页主视觉
2. 人物详情卡头图
3. 事件详情场景图
4. 朝代概览横幅图

#### 11.9.3 使用规则

1. 插图是辅助理解，不得压过图谱主体。
2. 数据缺失时，允许展示统一风格占位插图。
3. 同一对象在不同页面的插图口径应一致，避免认知冲突。

## 12. 权限边界、状态处理和边界条件

### 12.1 权限边界

一期默认所有用户均可浏览，无角色分层。

### 12.2 页面状态

1. 初始态：展示首页推荐与搜索入口。
2. 朝代首页态：展示该朝代的基础骨架和推荐切入点。
3. 节点聚焦态：图谱高亮当前人物或事件，并展开详情面板。
4. 加载态：切换朝代、首次加载图谱或搜索跳转时展示加载反馈。
5. 空态：当前朝代或对象数据不足时展示结构占位和说明。
6. 无结果态：搜索未命中时展示引导文案。
7. 失败态：加载失败时允许用户重试，并保留当前入口语境。

### 12.3 边界条件

1. 人物重名：必须通过朝代和身份标签区分。
2. 同一人物跨多个重要事件：详情面板应允许列出多事件入口。
3. 节点过密：优先展示焦点上下文，不强求全量同屏。
4. 数据缺失：允许缺字段，但不允许缺页面骨架。
5. 朝代跨度大：允许先以朝代内部探索为主，不要求一期完成跨朝代全局联通图。

## 13. 页面 / 模块状态矩阵

| 模块 | 默认态 | 交互态 | 空态 | 异常态 |

## 15. 反馈文案规则和提示时机

1. 数据不足时使用“资料持续补充中”，避免让用户误解为系统错误。
2. 搜索无结果时，提示用户换人物名、事件名或朝代名尝试。
3. 节点过多被折叠时，应提示“已聚焦核心关系，可继续展开查看”。
4. 图谱首次进入时，应给出简短交互提示，如“拖动画布查看关系，点击节点查看详情”。

## 16. 非功能要求

1. 主图谱区必须支持连续、顺滑的缩放和平移体验，不能退化为普通文档滚动页面。
2. 在节点和关系较多时，界面仍需保持清晰可操作，不因信息量稍大即失去可用性。
3. 首屏需建立明显的历史气质和探索感，不能呈现为普通资讯站或后台式列表页。
4. 一期只要求主流桌面浏览器中的完整体验，不对移动端和客户端形态做交付要求。

## 17. 验收标准

### 17.1 P0 验收

1. 用户可从首页进入任一朝代图谱。
2. 用户可通过搜索找到人物、事件、朝代，并进入对应上下文。
3. 用户可在图谱中完成拖拽、缩放、点击、聚焦、展开。
4. 用户点击人物后可看到人物详情，并继续跳转到相关人物和相关事件。
5. 用户点击事件后可看到事件详情，并高亮相关人物。
6. 当前朝代必须同时具备图谱、时间轴和详情面板三种信息层。
7. 页面存在完整的加载态、空态、无结果态和失败态。
8. 插图位已纳入页面结构，即使内容使用占位素材也可正常呈现。

### 17.2 不通过条件

1. 搜索只能跳到静态详情页，无法进入关系上下文。
2. 图谱只是静态图片或固定排版，无法交互探索。
3. 图谱缺少详情承接，用户点击后没有信息闭环。
4. 页面风格退化为普通文字列表站。

## 18. 需求追踪表

| 需求 | 优先级 | 对应模块 | 验收口径 |
|---|---|---|---|
| 朝代进入与切换 | P0 | 首页、朝代图谱 | 可进入任一朝代并切换 |
| 图谱探索 | P0 | 图谱主画布 | 可缩放、拖拽、聚焦、展开 |
| 搜索人物 / 事件 / 朝代 | P0 | 全局搜索 | 搜索命中后进入上下文 |
| 人物详情 | P0 | 人物详情面板 | 支持信息展示和继续探索 |
| 事件详情 | P0 | 事件详情面板 | 支持信息展示和关系高亮 |
| 时间轴联动 | P0 | 时间轴 | 点击事件可联动图谱 |
| 插图承接 | P0 | 首页、详情、概览 | 插图区结构可用且不喧宾夺主 |

## 19. 下游交接输入

### 19.1 给 UI

1. 核心页面不是单页静态排版，而是“图谱主画布 + 详情面板 + 时间轴 + 插图”的复合探索界面。
2. 需要优先定义三层信息密度：默认概览、节点聚焦、深度展开。
3. 必须覆盖首页、图谱页、搜索结果态、人物详情态、事件详情态、数据不足态、无结果态。
4. 插图与图谱的关系是“辅助记忆和氛围”，不可喧宾夺主。

### 19.2 给前端 / 架构

1. 主图谱是产品核心能力，必须按“大规模关系浏览场景”设计，不应按普通 DOM 列表思路承接。
2. 产品需要支持对象间多类型关系、按朝代切换、按搜索结果定位、按焦点收束信息密度。
3. 数据结构必须支持人物、事件、朝代、关系、插图占位这些对象长期扩展。
4. 页面必须允许数据不完整时正常运行，因此对象和关系都需要“待补全”状态。

## 20. 方案取舍说明

### 20.1 推荐主方案

采用“首页引导 + 朝代图谱主画布 + 右侧详情面板 + 底部或侧边时间轴 + 插图增强”的复合探索方案。

理由：

1. 既能满足你要求的图谱探索感，也能避免纯图关系难懂。
2. 搜索、详情、时间轴和插图都能找到稳定位置，不会彼此抢入口。
3. 更适合后续逐步补数据，而不是一次性做完整百科。

### 20.2 更简单备选方案

采用“时间轴 + 详情页 + 搜索”的轻量方案。

不推荐作为一期主方案，原因：

1. 无法体现关系网络优势。
2. 容易退化成普通历史内容站。
3. 难以承接你提出的 `canvas` / `WebGL` 级图谱体验诉求。

## 21. 风险与后续建议

### 21.1 主要风险

1. 如果一开始就要求全朝代全人物联通，范围会快速失控。
2. 如果图谱中关系类型过多、层级过深，页面会变成噪音。
3. 如果没有统一的节点信息最低结构，后续补数据时页面会反复返工。

### 21.2 建议的下一步

1. 先由 UI 输出首页和朝代图谱页的信息架构稿。
2. 再由前端 / 架构确定主图谱交互承接方式和数据骨架。
3. 数据整理阶段先选 1 到 2 个代表朝代做样板，再扩大覆盖范围。
